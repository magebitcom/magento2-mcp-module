Handing the AI a key, safely.

Magebit MCP is built so that connecting an AI assistant to your store doesn't expand what any admin could already do — and so every single call leaves a trail. Here's the plain-language version.

The shape of a token

A token is a secret string that identifies one admin user. You generate it from the command line, paste it into your AI client, and it never shows up again — the database keeps a bcrypt hash, not the plaintext. Revoking it invalidates the token immediately; the audit history it produced is preserved.

  • Plaintext shown once, at creation.
  • Stored as a bcrypt hash — a leaked database still can't be used to impersonate you.
  • Revoked tokens fail auth on the very next request.
  • Each token is owned by exactly one admin user.

Permissions follow the admin, not the AI

A token inherits the role of the admin who owns it. If that admin can't see orders in the Magento admin UI, the AI can't read them either. This is re-checked on every call — not just at login — so if you strip a role's permissions, everything the AI does with that token goes with it.

Tools that wrap a Magento service contract (like creating an invoice) also re-check the underlying admin-UI permission. In other words: MCP cannot do what the admin UI cannot do.

Writes are off by default

Read operations (lookups, reports, listings) are what most teams want from day one. Write operations (creating invoices, updating products, publishing CMS content) require two switches to both be on:

  1. A global kill-switch in Stores → Configuration → Magebit → MCP Server → Allow Write Tools.
  2. A per-token allow_writes flag, set when you mint the token.

Turn off either and every write is blocked with a clear error. You can have a read-only token for a curious colleague and a write-enabled one for your operations lead.

Every call is audited

Every POST /mcp — successful or not — writes one row to magebit_mcp_audit_log:

  • Who (admin + token)
  • What (tool name, arguments, result summary)
  • When (start time, duration in milliseconds)
  • How it ended (error code if it failed)

Sensitive fields — emails, phone numbers, addresses, the rest — are replaced by a 16-character HMAC fingerprint before the row is written. You can still tell two calls referred to the same email without ever seeing the email. The audit view lives in the Magento admin under a separate ACL resource, so only the roles you choose can read it.

A cron prunes rows older than your configured retention window — set it to 0 to keep forever.

Network-level defenses

Bearer auth gates most attacks, but two more layers sit in front:

  • Origin allowlist. The Origin header on every incoming request is matched against an admin-editable list. This blocks a class of attack called DNS rebinding, where an attacker tricks your browser into making a request from a hostile page. Defaults cover loopback plus the major AI clients.
  • 256 KiB body cap. The server refuses request bodies larger than 256 KiB before authentication runs, so unauthenticated attackers can't exhaust PHP worker memory with a flood.

The one-click emergency stop

If something feels wrong, there's a master kill-switch in Stores → Configuration → Magebit → MCP Server → Enable MCP Server. Turn it off and every request returns HTTP 503 immediately, before auth even runs. No code deploy, no downtime, no scramble — just one toggle.

Common questions

Can the AI place orders I haven't approved?

Not without the write switches both being on, and not unless the token's owning admin has the role for it. Many teams leave writes off entirely and use the AI purely for insight. If you do turn writes on, the module also asks the AI client to surface a confirmation prompt to the human before write tools run — Claude Desktop does this, and well-behaved clients respect it.

What happens if a token leaks?

Revoke it from the admin UI or the command line. The next request using that token fails auth with UNAUTHENTICATED. The audit rows the token produced while active stay in place — you'll see exactly what it did.

Where is the audit log?

In the Magento admin, under a dedicated section gated by its own ACL resource (Magebit_Mcp::mcp_audit). That means your token-manager role doesn't automatically get to read the audit log, and vice versa — you compose the roles to fit your team.

Does the AI see customer emails and phone numbers?

It can, if the tool the AI calls returns them and the admin role has permission to see them. What the AI sees in a live response is not what lands in the audit log: the log replaces those fields with a one-way HMAC fingerprint at write time.

How do I turn it off in an emergency?

Flip Stores → Configuration → Magebit → MCP Server → Enable MCP Server to No. Every request now returns HTTP 503 before auth runs. Alternatively, revoke the specific token from the CLI with bin/magento magebit:mcp:token:revoke.

Does the module expose any public endpoint?

One: POST /mcp. It requires a bearer token, a correct Mcp-Protocol-Version header, and an allowed Origin. There is no GET /mcp, no anonymous discovery, no form.
